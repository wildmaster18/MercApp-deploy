MercApp

Aplicación web para gestión y venta de productos

Mateo Carranza — Ingeniería en Software — Universidad Politécnica Salesiana

Resumen técnico

MercApp es una tienda en línea compuesta por una API REST y una SPA que trabajan juntas. El backend está construido con Node.js, Express y MongoDB, y expone endpoints para gestionar productos, categorías, autenticación de usuarios y un chat en tiempo real con Socket.io. El frontend es una Single Page Application desarrollada con Vue 3, Vue Router y Pinia, que consume la API y presenta toda la interfaz sin recargar la página.

El proyecto corresponde a la Unidad 4 de Aplicaciones Web e integra lo aprendido en las cuatro unidades del curso: desde los fundamentos web y el backend con Node.js, hasta la SPA con Vue 3 y el despliegue en la nube sobre MongoDB Atlas, Railway, Netlify y GitHub Pages.

Arquitectura de despliegue

Usuario Navegador web
→
Netlify SPA Vue 3 (dist/) HTTPS + CDN
→
Railway API REST Express Helmet + Rate Limit
→
MongoDB Atlas Cluster M0 TLS activo

El usuario abre la SPA servida por Netlify mediante HTTPS. La SPA consume la API en Railway, también por HTTPS, y CORS restringe el acceso a los dominios configurados. La API guarda los datos en MongoDB Atlas con conexión mongodb+srv y TLS. El micrositio de documentación en GitHub Pages es independiente del flujo de datos y solo publica esta información del proyecto.

Sobre la base de datos: se encuentra en MongoDB Atlas con conexión mongodb+srv y TLS. En este despliegue se documenta una allowlist amplia temporal (0.0.0.0/0) debido a la limitación de IP fija en el plan gratuito de Railway. Es una decisión temporal y documentada. El acceso queda protegido por las credenciales de Atlas, las variables de entorno y la configuración segura del backend.

Las rutas marcadas con "Sí" exigen una sesión activa mediante un middleware en el backend y responden 401 sin sesión. Las rutas de lectura quedan públicas para facilitar la revisión del API. /api/auth/me no rechaza la petición: devuelve los datos del usuario si hay sesión o null si no la hay, y es lo que usa el frontend para verificar la sesión real.

Variables de entorno

Backend (Railway, producción)

Variable Descripción
MONGODB_URI Cadena mongodb+srv del cluster de Atlas
SESSION_SECRET Clave para firmar las sesiones, obligatoria en producción
FRONTEND_URL Origen permitido en CORS (dominio de Netlify)
NETLIFY_URL Origen adicional permitido en CORS
NODE_ENV Se fija en production para activar la cookie segura
PORT Lo asigna Railway automáticamente (process.env.PORT)

Frontend (build de Netlify)

Variable Descripción
VITE_API_URL URL pública del API en Railway que consume la SPA

El archivo .env.local es solo local y no se versiona. La plantilla a publicar es .env.example, sin valores reales. Los secretos nunca se suben al repositorio y SESSION_SECRET es obligatorio en producción. El puerto lo asigna Railway.

Seguridad y configuración

  • HTTPS activo en Netlify, Railway y GitHub Pages.
  • CORS restringido por FRONTEND_URL y NETLIFY_URL; no se usa el comodín *.
  • Cookies de sesión con secure y sameSite: none en producción para el uso cross-domain.
  • El frontend envía credentials: include en las peticiones que usan sesión.
  • Verificación de sesión real con GET /api/auth/me; el router no depende solo de localStorage.
  • Rutas de escritura de productos y categorías protegidas con sesión.
  • SESSION_SECRET obligatorio cuando NODE_ENV=production.
  • Helmet para cabeceras de seguridad HTTP.
  • express-rate-limit para limitar las peticiones al API.
  • express-validator para validar y sanitizar los datos de entrada.
  • MongoDB Atlas con conexión mongodb+srv y TLS.
  • Archivos .env reales fuera del control de versiones; .env.local solo local.

Guía de ejecución local

Requisitos

Node.js v24 recomendado y MongoDB local o MongoDB Atlas configurado.

Backend

cd backend
npm install
npm run seed
npm run dev

El servidor queda disponible en http://localhost:3000. El seed crea categorías, productos de prueba y un usuario admin/admin123.

Frontend

cd frontend
npm install
npm run dev

La SPA queda disponible en http://localhost:5173.

Build de producción

cd frontend
npm run build
npm run preview

El build genera la carpeta dist/ que se publica en Netlify. Antes de construir se define VITE_API_URL con la URL del API en Railway.

Pruebas de verificación

Pruebas sugeridas para comprobar el despliegue:

  • Abrir el frontend publicado en Netlify.
  • Iniciar sesión con admin / admin123.
  • Consultar /health y /api/health.
  • Ver el listado de productos.
  • Crear un producto.
  • Editar un producto.
  • Eliminar un producto.
  • Crear una categoría.
  • Agregar productos al carrito y finalizar la compra.
  • Cerrar sesión.
  • Recargar rutas internas de Vue Router sin obtener 404.
  • Verificar que el micrositio carga en GitHub Pages.

Decisiones y ajustes de despliegue

  • Railway asigna el puerto, por eso el backend lo lee desde process.env.PORT.
  • Railway gratuito no ofrece IP fija de salida, por eso Atlas usa una allowlist amplia temporal y documentada.
  • Netlify necesita el archivo _redirects para que Vue Router no devuelva 404 al recargar una ruta interna.
  • Las cookies cross-domain requieren credentials: include en el frontend y sameSite: none con secure en el backend.
  • Los archivos .env reales no se versionan.
